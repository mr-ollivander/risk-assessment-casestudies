Risk Assessment · Case Study 01

Malicious File Upload via
Unauthenticated Contact Form

A walkthrough of how I approached a real-world file upload vulnerability — from initial scoping questions through to a rated risk and a concrete response plan.

Technical Infrastructure
AWS S3 · File Upload · RCE
Overall Risk: High
The Scenario
What we're working with

A retail company runs a customer-facing e-commerce website with a contact form that routes enquiries to the support team's shared inbox. The development team recently added a file upload feature so customers can attach screenshots of order issues.

Key technical details that drove this assessment:

  • The upload feature accepts any file type — no validation in place
  • Uploaded files land in a publicly accessible S3 bucket
  • The bucket URL follows a predictable pattern: company-support-uploads-{year}
  • No malware scanning on uploaded files
  • The support team opens attachments directly from the bucket URL in their browsers
The Thinking
How I reasoned through it
First instinct

My first question was about the shared inbox access model — should junior support staff have visibility into all tickets, or is there tiered access for sensitive issues? I wanted to understand whether the inbox itself was an asset worth protecting separately.

Considered and rejected

I tabled the inbox access question once I recognized the scenario doesn't define any tiered classification. Contact forms are generally L1-accessible by design — escalation happens after intake, not before. That question needed more context than the scenario provided, so I moved on.

Assumption revised

My initial threat model assumed an authenticated customer as the attacker — someone already logged in. But I reconsidered: contact forms typically don't require authentication. Anyone can reach them. That removes the only friction point between an attacker and the upload functionality entirely.

Key insight

There are two vulnerabilities, not one — and they have a dependency direction. I initially thought the public S3 bucket enabled the file upload flaw. Wrong. The validation gap has to exist first for anything malicious to reach the bucket at all. Sequence matters in attack chains.

On impact scope

The asset isn't customer data in screenshots — it's company systems. Realistically, it's the support team's workstations that open these files. RCE via a malicious upload could theoretically chain into lateral movement, but that increases attack complexity significantly. Impact rated High, not Critical, on that basis — I didn't inflate the rating with a theoretical worst case.

Don't forget the past

The fix prevents future uploads. But files already in that bucket need to be addressed too. They can't simply be opened and checked — they may be malicious. A malware scan of existing bucket contents is a required step before any access continues.

The Assessment
Formal risk evaluation
Asset
Company systems and servers — primarily support team workstations that open uploaded files. Lateral movement to internal infrastructure is a theoretical escalation path but increases attack complexity.
Threat
An unauthenticated external attacker submitting a malicious file via the publicly accessible contact form. No authentication barrier exists between attacker and upload functionality.
Vulnerability
Lack of server-side file upload validation (root cause) enabling malicious file upload — compounded by a publicly accessible S3 bucket with a predictable naming pattern that broadens the attack surface and enables enumeration.
Likelihood
HIGH  No authentication, no file validation, no malware scanning, predictable bucket URL, support team opens files directly in browser. Low attack complexity end-to-end.
Impact
HIGH  RCE affecting confidentiality, integrity, and availability of company systems. Lateral movement possible but requires attacker knowledge of internal infrastructure — not assumed in base rating.
Overall Risk
HIGH  High likelihood × High impact.

The Response
What I'd recommend, and in what order
01
Scan and remediate existing bucket contents
Before any other action, existing files in the S3 bucket must be scanned for malware. They cannot simply be opened to check — they may be malicious. Confirmed malicious files should be removed immediately. This addresses the current state, not just future uploads.
02
Implement server-side file upload validation
Extension allowlisting, MIME type checking, and heuristic/AV scanning on every upload. Client-side validation alone is trivially bypassed — Burp Suite, curl, any proxy intercept — and should be treated as a UX aid only, not a security control. Server-side is the only validation that closes the gap.
03
Restrict S3 bucket access and fix URL predictability
Remove public access from the bucket. Scope access to the support team and relevant customers only via IAM policies. Introduce sufficient entropy in bucket naming to prevent enumeration — predictable patterns like company-support-uploads-{year} are trivial to brute-force systematically.
