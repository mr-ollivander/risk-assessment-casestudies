Case Study 01 · Technical Infrastructure

Malicious File Upload via
Unauthenticated Contact Form

A walkthrough of how I approached a real-world file upload vulnerability — from initial scoping questions through to a rated risk and a concrete response plan.

AWS S3 · File Upload · RCE
Overall Risk: High
Response: Mitigation
The Scenario
What we're working with

A retail company runs a customer-facing e-commerce website with a contact form that routes enquiries to the support team's shared inbox. The development team recently added a file upload feature so customers can attach screenshots of order issues.

Key technical details:

  • The upload feature accepts any file type — no server-side validation in place
  • Uploaded files are stored in a publicly accessible S3 bucket
  • Bucket URL follows a predictable pattern: company-support-uploads-{year}
  • No malware scanning on uploaded files
  • The support team opens attachments directly from the bucket URL in their browsers
The Thinking
How I reasoned through it
First instinct

My first question was about inbox access control — should junior support staff have visibility into all tickets, or is there tiered access for sensitive issues? I wanted to know whether the inbox itself was a separate asset worth scoping.

Considered and tabled

I tabled the inbox question when I realised the scenario doesn't define tiered classification. Contact forms are generally L1-accessible by design — escalation happens after intake, not before. Moving on.

Assumption revised

My initial threat model assumed an authenticated user as the attacker. But contact forms typically require no login. Anyone can reach them. That removes the only friction point between an attacker and the upload functionality entirely.

Key insight — two vulnerabilities, one dependency

There are two distinct vulnerabilities, and the direction of dependency matters. I initially assumed the public S3 bucket enabled the file upload flaw. Wrong. The validation gap must exist first — a malicious file has to get onto the server before the bucket's public access can amplify the damage. Sequence matters in attack chains.

On asset and impact scope

The real asset is company systems — specifically the support team's workstations that open these files. RCE via a malicious upload could theoretically chain into lateral movement across internal infrastructure, but that requires the attacker to already know the internal network topology. Impact rated High, not Critical — I don't inflate ratings with theoretical worst cases that aren't supported by the scenario.

Don't forget the past

The fix addresses future uploads. But files already sitting in that bucket are unresolved — they can't simply be opened to check, because they may already be malicious. Scanning existing bucket contents is a required step before any access continues, not an afterthought.

The Assessment
Formal risk evaluation
Asset
Company systems and servers — primarily support team workstations that open uploaded files. Lateral movement to broader internal infrastructure is a theoretical escalation path that increases attack complexity.
Threat
An unauthenticated external attacker submitting a malicious file via the publicly accessible contact form. No authentication barrier exists between attacker and the upload functionality.
Vulnerability
Lack of server-side file upload validation (root cause) enabling malicious file upload — compounded by a publicly accessible S3 bucket with a predictable naming pattern that broadens the attack surface and enables enumeration.
Likelihood
HIGHNo authentication barrier, no file type validation, no malware scanning, predictable bucket URL pattern, support team opens files directly in browser. Low attack complexity end-to-end.
Impact
HIGHRCE affecting confidentiality, integrity, and availability of company systems. Lateral movement is possible but requires knowledge of internal infrastructure — not assumed in the base rating.

The Response
What I'd recommend, and in what order
01
Scan and remediate existing bucket contents
Before anything else, existing files in the S3 bucket must be scanned for malware — they cannot be opened to check, as they may already be malicious. Confirmed malicious files should be removed immediately. This addresses the current state, not just future uploads.
02
Implement server-side file upload validation
Extension allowlisting, MIME type checking, and heuristic or AV scanning on every upload. Client-side validation is trivially bypassed via any proxy intercept — it is a UX aid only, not a security control. Server-side validation is the only mechanism that actually closes the gap.
03
Restrict S3 bucket access and eliminate URL predictability
Remove public access from the bucket. Scope access via IAM policies to the support team and relevant customers only. Introduce sufficient entropy in bucket naming — patterns like company-support-uploads-{year} are trivial to enumerate systematically.
